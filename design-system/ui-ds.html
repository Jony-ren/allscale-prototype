10 · Responsive

    
Breakpoints

    
Named modal responsive laws (W0xxA). Each modal declares one breakpoint at 633px — desktop ≥634px, compact ≤633px — and drops the canonical back/close header for a centered title. No 768px sheet fallback. Reference: checkout/checkout-onramp.html.


    


      

        
W002APrimary modal → fullscreen, keep Back

        
Card & Local Payments · single breakpoint 633px

        

          

            
≥ 634px · Desktop

            
    
              
  • Top-centered modal · 420px wide · radius 24px
    • 
              
  • Banner: title only — no back, no close
    • 
              
  • Confirm / Cancel footer at the modal bottom (position:sticky)
    • 
            

          

          

            
≤ 633px · Fullscreen

            
    
              
  • Fills viewport · 100% × 100% · radius 0
    • 
              
  • Back button shown (only nav kept); close stays hidden
    • 
              
  • Title stays centered via invisible close-spacer
    • 
              
  • CTA pinned to the screen bottom (flex:1 body + footer margin-top:auto)
    •

                    background: var(--accent-light); border-radius: var(--radius-full); }

Rules

      
    
        
  • Back button always left-anchored, close button always right-anchored
    • 
        
  • Both icon buttons are 44×44 (var(--icon-btn)) circular, accent-light bg
    • 
        
  • SVG inside is 18×18, stroke-width 2.2, color var(--text)
    • 
        
  • If a step has no Back, render an invisible spacer with visibility:hidden to keep close on the right
    • 
        
  • Modal frame width is fixed at 420px (var(--modal-w)) across every step of a sub-flow — no width-jitter between screens

               border: 1.5px solid var(--border);
               border-radius: var(--radius-md);
               box-shadow: var(--shadow-popover);
               padding: 4px; }

               font-size: var(--text-sm); font-weight: var(--weight-medium);
               border-radius: var(--radius-xs); }
.ds-dd-item:hover { background: var(--accent-light); }
.ds-dd-item.danger { color: var(--red); }

      
Rules

      
    
        
  • Container: 140px min-width (var(--dropdown-min-w)), 1.5px border, 12px radius, 4px inner padding, var(--shadow-popover) elevation
    • 
        
  • Items: 10×14 padding, 14px medium text, 8px radius (so hover bg stays inside the menu's rounded edge)
    • 
        
  • Hover: var(--accent-light) background for every item, including destructive — colour difference comes from text only
    • 
        
  • Destructive items: red text (var(--red)), no extra red background — keep hover language consistent
    • 
        
  • Position: absolute from the trigger button, 6px vertical gap, anchored to the trigger's side (usually right)
    • 
        
  • Do NOT set overflow: hidden on the menu — items already round their own corners
    •

/* Spec */
.ds-toast { background: var(--surface); border: 1px solid var(--border);
            border-radius: var(--radius-2xl); padding: 12px 18px;
            font-size: 15px; font-weight: var(--weight-medium);
            color: var(--text); letter-spacing: var(--tracking-snug);
            box-shadow: var(--shadow-lg); display: inline-flex;
            align-items: center; gap: 10px; }

/* Positioning + show/hide */
.toast-live { position: fixed; left: 50%; bottom: 28px; z-index: var(--z-toast);
              transform: translateX(-50%) translateY(12px); opacity: 0;
              transition: opacity var(--dur-base) var(--ease-out),
                          transform var(--dur-base) var(--ease-out); }
.toast-live.show { opacity: 1; transform: translateX(-50%) translateY(0); }

      
Rules

      
    
        
  • Surface fill (var(--surface)), 1px border, 28px radius (var(--radius-2xl)), 12×18 padding, var(--shadow-lg) elevation
    • 
        
  • Text: 15px medium, var(--text), snug tracking; optional leading icon/dot with 10px gap
    • 
        
  • Position: fixed, bottom-center, 28px from bottom, var(--z-toast) (1000)
    • 
        
  • Motion: slide up 12px + fade in over var(--dur-base) with var(--ease-out); auto-dismiss after ~2.5s
    • 
        
  • One toast at a time. Keep copy to a short outcome (“Saved”, “Wallet address saved”) — not errors that need action

00 · Philosophy

    
Users shouldn't have to think.

    
Our job isn't to make AllScale beautiful — it's to make it intuitive, obvious, trustworthy, and efficient. Every decision should reduce cognitive load.


    

      

        
01 · CLARITY OVER CREATIVITY

        
Obvious beats clever.

        
Simple beats impressive; familiar beats novel. If a user has to stop and ask what does this mean? the design has failed.

      

      

        
02 · MENTAL MODELS

        
Recognition over recall.

        
Users should instantly know where they are, what they can do, and what happens next. Reach for established patterns — never hidden functionality.

      

      

        
03 · ONE PRIMARY ACTION

        
One goal per screen.

        
Define the user goal, the business goal, and a single primary action. Remove anything that competes with it.

      

      

        
04 · RUTHLESS SIMPLICITY

        
Can this be removed?

        
Reduce choices, steps, fields, and clicks. Always ask that before asking “can this be improved?”

      

      

        
05 · SAY WHAT HAPPENS

        
Label the outcome.

        
Buttons and copy describe exactly what happens — Pay Invoice, not Continue. Users scan; give them hierarchy, not jargon.

      

      

        
06 · TRUST OVER DELIGHT

        
Money is always explicit.

        
Communicate balances, fees, status, and risk clearly. Never hide financial information — users must know what happened, what is happening, and what will happen.

      

    


    

      Users do not want to think. The moment someone stops to ask “what do I click?” — the design has failed.
